# freekoko-app вЂ” P2 Notes

This document captures context for the P3/P4/P5/P6 agents who will build on
the Electron shell produced in P2.

## What P2 delivered

- Electron 34 main process, preload, and renderer scaffold wired via
  `electron-vite` (config file: `electron.vite.config.ts`).
- `SidecarSupervisor` with full lifecycle: spawn, health polling with
  exponential backoff (250ms в†’ 2s, capped at 30s), SIGTERM в†’ SIGKILL
  stop, crash restart with the specified backoff `[500, 1000, 2000, 5000,
  10000]` capped at 5 attempts per 60s window, orphan SIGKILL on
  `process.on('exit')`, port-in-use detection from sidecar log lines (R9).
- `SidecarClient` with Node `fetch`, timeouts (3s health/voices, 30s TTS).
- `LogCapture` with newline-delimited JSON parsing, ring buffer (1000),
  file forwarding via `electron-log`, broadcast to renderer on `on:log-line`.
- `SettingsStore` typed wrapper around `electron-store` with in-process
  `changed` EventEmitter event wired to `on:settings-changed`.
- `TrayMenu` with idle/starting/running/stopping/crashed/port_in_use/error
  menus and three placeholder icon variants (22Г—22 + @2x).
- All 25 IPC channels from ARCHITECTURE В§3.3 exposed via preload. Real
  implementations for supervisor, settings, logs, window, app. Stubs for
  `tts.*` and `history.*` returning `{ error: 'not_implemented', phase: 'P3|P4' }`.
- Renderer shell: React 19 + Zustand, tab navigation, `StatusBadge`,
  placeholder views that pass the "Logs view shows real sidecar stdout in
  real time" acceptance criterion via a subscription in `useSidecar`.
- Platform guard: non-darwin or non-arm64 shows `dialog.showErrorBox` and
  quits.
- Single-instance lock, menubar-first (`app.dock.hide()`, `window-all-closed`
  does not quit), lazy main window created only on demand.
- `before-quit` awaits `supervisor.stop({graceful:true})` before re-issuing
  `app.quit()` via `setImmediate`.

## Deviations from the spec

1. **electron-vite 3.1.0 instead of 2.3.0**. Required because Vite 6 (mandated
   by the spec) conflicts with electron-vite 2.x's `peer vite@^4 || ^5`.
   electron-vite 3.1.0 supports Vite 4/5/6. No API changes needed.

2. **Config filename is `electron.vite.config.ts` (dots)**, not
   `electron-vite.config.ts`. The electron-vite CLI looks up
   `electron.vite.config.{js,ts,mjs,cjs}` by convention.

3. **TypeScript `tsconfig.node.json` uses `module: ESNext` + `Bundler`**
   rather than CommonJS. This is required because `electron.vite.config.ts`
   uses `import.meta.url` to compute `__dirname` equivalents (ESM), and
   TypeScript's `import.meta` support requires ES2020+ module target.
   Source files are emitted as CJS by `electron-vite` regardless; the
   tsconfig only governs typechecking.

4. **Tray icons are solid-color circles generated by
   `scripts/generate-tray-icons.mjs`**. Valid 22Г—22 and 44Г—44 PNGs with the
   correct bytes for each state (grey/green/red). P6 agent should replace
   with real microphone art.

5. **`outputDir` default is `userData/history`**, not `~/Music` as
   the ARCHITECTURE.md В§3.5 suggests. The user's P2 spec explicitly says
   `app.getPath('userData') + '/history'` and we followed that spec. P5
   agent may reconcile if needed.

6. **Open-app-via-tray deep links to specific tabs** (Settings/Logs) go
   through a custom broadcast channel `on:navigate` that is NOT currently
   subscribed by the preload. The renderer App.tsx has a placeholder
   effect but the channel is not yet exposed on `window.electronAPI`.
   P5 agent should finalize the navigation plumbing.

## Known gaps / items for later agents

- **No end-to-end `npm run dev` run.** The sidecar binary does not yet exist
  (P1 is building it in parallel). `SidecarSupervisor.start()` correctly
  handles ENOENT and transitions to an `error` state with the message
  "Sidecar binary not found вЂ” run `make sidecar` to build it."
  Verify interactively once P1 ships.
- **`tts.*` and `history.*` IPC handlers are stubs.** P3/P4 will replace
  with real implementations and wire the `on:tts-progress` emitter.
- **No `HistoryStore`** yet вЂ” directory exists but empty. P3 creates it
  (index.json + WAV files).
- **No voice dropdown, no generate button, no audio player** вЂ” P3.
- **`LogsView` is a minimal scrolling list** вЂ” P4 should add CSS virtual
  scroll and auto-scroll-pause.
- **Settings view is a placeholder** вЂ” P5 builds the form with port,
  default voice, default speed, auto-start, launch-at-login.
- **No macOS version guard** (Darwin 24 check). P5 acceptance criterion.
- **Sandboxed preload warning:** `sandbox: false` is set on the main window
  so the preload can use Node imports freely. Acceptable for P2; P6 may
  revisit when tightening CSP / supply-chain posture.
- **electron-builder config** is not yet written (`electron-builder.yml`).
  P6 adds it together with the DMG workflow.
- **CSP** in `src/index.html` allows `unsafe-inline` for scripts and styles.
  Tighten in P6.
- **`npm audit`** reports vulnerabilities inherited from transitive deps.
  None are direct dependencies; review during P6 packaging.

### Deviations / interop notes

- **P3 extended `src/lib/ipc.ts`** with the real `history.*` wrappers
  (`listHistory`, `deleteHistory`, `saveHistoryWav`, `clearHistory`,
  `readHistoryWav`). My HistoryView/HistoryItem imports those wrappers
  rather than talking to `window.electronAPI.history.*` directly, so
  if the preload channel shape changes further, only `ipc.ts` needs a
  tweak.
- **`useHistoryStore` is still the stub shape from P2** (`items`,
  `setItems`, `isLoading`). HistoryView drives the list through IPC +
  `setItems`. Once P3 finalizes richer store actions, HistoryView can
  be trimmed down.
- **Voice flag map is duplicated** in `src/lib/format.ts` because
  GenerateView (P3) has its own. The brief explicitly permits this to
  avoid a circular dep; reconcile in a cleanup phase when both views
  are stable.
- **`window.confirm` is used** for Clear All and per-item Delete. The
  P4 spec mentioned `dialog.showMessageBox` via IPC as an alternative;
  sticking with `confirm` keeps the UI renderer-only and avoids adding
  another IPC channel mid-phase.

### What P3 / P5 need to follow up on

- `history.saveWav` handler should show the main-process save dialog
  when `destPath` is absent (the HistoryItem calls `saveHistoryWav(id)`
  with no path and treats `{canceled:true}` as a non-error).
- `history.readWav` handler needs to actually return `Uint8Array`
  bytes. Our client accepts `Uint8Array | ArrayBuffer | number[] |
  {bytes}|{data}|{type:'Buffer', data:[...]}` to be robust to IPC
  serializers.
- P3 may decide to ship `useHistoryStore.loadHistory()` / `append()` /
  `remove()` actions вЂ” HistoryView will happily switch to them.
- `role="log" aria-live="polite"` is applied on the LogsView's
  VirtualList container; if P5 adds a full-window screen-reader audit,
  the "N new" pill also doubles as an ARIA announcement trigger.
